# BJJManager — Master Specification

Status: draft v1 — produced from the system-building requirements in `ProjectIdea.md.txt`. This is
the single source of truth for the domain, API contract, and cross-cutting decisions. The backend,
frontend, and database execution plans each implement a slice of this document — they must not
redefine entities, endpoints, or rules, only reference them.

Internal project name in the idea doc is "BJJ Tracker"; the repo/solution is named `BJJManager`.
Both refer to the same product.

---

## 1. Scope

In scope, mapped onto BJJManager:

| Requirement | How BJJManager satisfies it |
|---|---|
| DB with ≥1 data table + 1 users table, PK + ≥2 fields | Users, TrainingSessions, TrainingMedia, Techniques, TechniqueSteps, TechniqueMedia (see §3) |
| Web API with CRUD, correct verbs/params/returns | REST API over Trainings and Techniques (see §4) |
| Second API surface: register, login, authorized + non-authorized endpoints | `/api/auth/register`, `/api/auth/login` are anonymous; everything else requires JWT |
| Data access layer, CRUD | EF Core repositories, Infrastructure layer |
| Business logic layer, independent of data layer and API | MediatR command/query handlers in Application layer, depend only on abstractions |
| Unit tests across data/business/API layers | xUnit test projects per layer, TDD-first (see backend plan) |
| Frontend framework, responsive, CRUD, structured code | Angular 18 (see frontend plan) |
| README + seeded demo data/credentials | Root README (later prompt) + automatic startup seeding (see database plan) |

## 2. Modules

1. **Authentication** — register (name + password only), login, JWT issuance.
2. **Training Sessions** — CRUD + filtering by exact date / month / year; each session has optional
   media metadata (no file upload — URLs only).
3. **Techniques Library** — CRUD; each technique has ordered steps (≥1, first step mandatory) and
   optional media metadata.
4. **Dashboard** — welcome message + totals.

Every record in modules 2–3 is strictly scoped to its owning user. No sharing, no admin role.

## 3. Domain model

### 3.1 User
| Field | Type | Rules |
|---|---|---|
| Id | Guid (PK) | generated by the entity on creation |
| Name | string | required, max 100 chars, **not** unique |
| PasswordHash | string | never exposed via API/DTO |
| CreatedAt | DateTime (UTC) | set on creation |

Password business rule: minimum 6 characters (enforced pre-hash, in the Application layer
validator — never in the entity, since the entity never sees the raw password after hashing).

### 3.2 TrainingSession
| Field | Type | Rules |
|---|---|---|
| Id | Guid (PK) | |
| UserId | Guid (FK → User) | owner, immutable after creation |
| TrainingDate | DateOnly/Date | required |
| AcademyName | string | required, max 200 chars |
| Notes | string? | optional |
| SelfEvaluation | int (1–5) | required; see enum below |
| CreatedAt | DateTime (UTC) | set on creation, immutable |

`SelfEvaluation` enum (`Application`/`Domain` layer, stored as tinyint):
`VeryBad = 1, BelowExpectations = 2, Average = 3, Good = 4, Excellent = 5`.

### 3.3 TrainingMedia
| Field | Type | Rules |
|---|---|---|
| Id | Guid (PK) | |
| TrainingId | Guid (FK → TrainingSession, cascade delete) | |
| FileName | string | required |
| FileUrl | string | required, stored as plain string (no upload logic — out of scope, future Firebase Storage integration) |
| MediaType | enum `Image \| Video` | required |

A training may have zero or many media entries.

### 3.4 Technique
| Field | Type | Rules |
|---|---|---|
| Id | Guid (PK) | |
| UserId | Guid (FK → User) | owner, immutable |
| Name | string | required, max 200 chars |
| Position | string | required, max 100 chars. Free text, not a closed enum — the spec lists positions ("Closed Guard", "Half Guard", "Side Control", "Mount", "Back Control", "Standing") as *examples*, not an exhaustive set. Frontend offers these as a dropdown with a "custom" free-text option; backend does not validate against a fixed list. |
| Description | string | required |

### 3.5 TechniqueStep
| Field | Type | Rules |
|---|---|---|
| Id | Guid (PK) | |
| TechniqueId | Guid (FK → Technique, cascade delete) | |
| Order | int | 1-based, unique per TechniqueId, contiguous |
| Description | string | required |

Invariant (enforced in the Application layer, validated against the Domain aggregate): a
technique must always have at least one step; step 1 is mandatory and can never be removed if it
is the only remaining step. Reordering on add/remove is the API's responsibility — the client
sends the full ordered step list on update, the handler re-numbers `Order` sequentially.

### 3.6 TechniqueMedia
Same shape as TrainingMedia, but FK is `TechniqueId → Technique` (cascade delete). Media belongs
to the technique, never to an individual step.

## 4. API contract

Base route: `/api`. All responses are JSON. All non-auth endpoints require
`Authorization: Bearer <token>` and are implicitly scoped to the authenticated user
(`UserId` is taken from the JWT claims, never from the request body/query for ownership checks).

Ownership violation policy: if a record exists but belongs to another user, the API returns
**404 Not Found** (not 403) — existence of another user's record is not disclosed.

### 4.1 Auth (anonymous)
| Verb | Route | Body | Success | Errors |
|---|---|---|---|---|
| POST | `/api/auth/register` | `{ name, password }` | 201 `{ id, name }` | 400 validation |
| POST | `/api/auth/login` | `{ name, password }` | 200 `{ token, expiresAtUtc, userName }` | 401 invalid credentials |

### 4.2 Trainings (authenticated)
| Verb | Route | Notes |
|---|---|---|
| GET | `/api/trainings?date=&month=&year=` | all filters optional and mutually exclusive-ish (date takes precedence if combined); returns only caller's records |
| GET | `/api/trainings/{id}` | 404 if missing/not owned |
| POST | `/api/trainings` | body excludes Id/UserId/CreatedAt; 201 + Location header |
| PUT | `/api/trainings/{id}` | full replace of editable fields; 204 |
| DELETE | `/api/trainings/{id}` | cascades media; 204 |
| POST | `/api/trainings/{id}/media` | add one media metadata entry; 201 |
| DELETE | `/api/trainings/{id}/media/{mediaId}` | 204 |

### 4.3 Techniques (authenticated)
| Verb | Route | Notes |
|---|---|---|
| GET | `/api/techniques` | list, caller's only |
| GET | `/api/techniques/{id}` | includes ordered steps + media |
| POST | `/api/techniques` | body includes ≥1 step; 201 |
| PUT | `/api/techniques/{id}` | replaces name/position/description + full step list (add/remove/reorder in one call); 204 |
| DELETE | `/api/techniques/{id}` | cascades steps + media; 204 |
| POST | `/api/techniques/{id}/media` | 201 |
| DELETE | `/api/techniques/{id}/media/{mediaId}` | 204 |

### 4.4 Dashboard (authenticated)
| Verb | Route | Returns |
|---|---|---|
| GET | `/api/dashboard` | `{ userName, totalTrainings, totalTechniques }` |

## 5. Cross-cutting decisions

- **Backend**: .NET 8, ASP.NET Core Web API, MediatR (commands/queries), EF Core, FluentValidation
  in a MediatR pipeline behavior, Clean Architecture (Domain / Application / Infrastructure / WebApi).
- **Frontend**: Angular 18, standalone components, functional route guards + HTTP interceptor for JWT.
- **Database**: SQL Server (connection string already provided — see repo's
  `Chaves e StringConnections.txt`, not committed to source control as-is; real secrets go through
  `appsettings.Development.json`/user-secrets).
- **Auth**: JWT (HS256), no refresh tokens (out of scope) — token expiry only, matches "simple, not
  complex."
- **Password hashing**: ASP.NET Core Identity's `PasswordHasher<T>` used standalone (PBKDF2), *not*
  the full Identity membership system — avoids pulling in unneeded tables/complexity for a
  name+password scheme where names aren't unique.
- **IDs**: `Guid`, generated application-side on entity creation (not DB `NEWID()`), consistent with
  aggregates owning their identity.
- **Media**: metadata only, no upload pipeline. `FileUrl` is a plain string produced by uploading
  the file client-side to Firebase Storage (config in `Chaves e StringConnections.txt`); the API
  never receives or stores the binary file itself.
- **Testing**: TDD across all layers; xUnit + FluentAssertions + NSubstitute (backend),
  Jasmine/Karma (frontend, Angular CLI default).
- **No code comments** — enforced across all future implementation prompts.
- **Repository layout**: monorepo, single git repository at the project root (see plans for folder
  layout — `/backend`, `/frontend`, `/database`, plus this `/specs` folder and the eventual root
  README).
